### Important Note on npm audit vs pnpm Reality

`npm audit` was used as an approximation (pnpm audit requires network access to the advisory database). However, **the pnpm lockfile shows many reported vulnerabilities are already resolved**:

| Package | npm audit says | pnpm-lock.yaml actual | Verdict |
|---------|---------------|----------------------|---------|
| linkifyjs | <4.3.2 vulnerable | 4.3.2 installed | **False positive** — already fixed |
| immutable | 4.0.0-rc.1–4.3.7 | 4.3.8 installed | **False positive** — already fixed |
| postcss | vulnerable range | 8.5.12 installed | **False positive** — already fixed |
| markdown-it | 13.0.0–14.1.0 | 14.1.1 installed | **False positive** — already fixed |

This happens because `npm audit` doesn't understand pnpm's symlinked `node_modules/` structure. It may report old versions from hoisted/nested paths that are not actually used at runtime.

**Lesson:** Always cross-reference `npm audit` findings against `common/temp/pnpm-lock.yaml` before acting. The lockfile is the source of truth for what Rush/pnpm actually installs.
